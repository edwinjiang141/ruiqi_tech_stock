# 创业板科技股优质股票研究推荐系统 V1.2 完整开发计划（纯量化版）

> 制定日期：2026-05-23  
> 适用版本：V1.2（纯量化）  
> 目标周期：30 个工作日（6 周，按 5 天/周）  
> 交付目标：完成“可运行、可复盘、可运维”的生产可用最小系统

---

## 1. 计划目标与范围

### 1.1 总目标
在不引入 AI 信息源、公告爬取、LangGraph、Redis、Celery、Docker 的前提下，落地一套以 Tushare 结构化数据为核心的数据驱动型股票研究推荐系统，形成以下闭环：

```text
数据采集 → 数据治理 → 因子计算 → 综合评分 → 推荐输出 → 模拟交易 → 复盘评估 → 权重迭代
```

### 1.2 范围确认（必须做）
1. 股票池（创业板 + 科创板 + 科技属性行业）构建。  
2. Tushare 数据采集（行情、财务、估值、资金流、行业分类）。  
3. PostgreSQL 四层数据建模（ods/dwd/dm/sys）。  
4. 六大因子 + 风险扣分 + 最终评分。  
5. 推荐等级输出（S/A/B/C/D）与推荐解释模板化。  
6. FastAPI 后端 API + Vue3 前端展示页面。  
7. APScheduler 定时任务链路。  
8. 模拟交易与复盘（收益、回撤、超额、Rank IC）。  
9. 非 Docker 部署（venv + systemd + Nginx）。

### 1.3 范围排除（明确不做）
- 新闻、舆情、公告文本采集和分析。  
- 任何 LLM/Agent 推理链路。  
- Redis/Celery/Docker 相关基础设施。

## 4. 六周冲刺实施计划（逐周）

## 第 1 周：工程基线 + 数据库骨架
**目标**：完成可开发、可迁移、可运行的底座。  
**任务**：
1. 初始化 backend/frontend 项目与依赖。  
2. 建立 SQLAlchemy 模型与 Alembic 迁移框架。  
3. 完成 ods/dwd/dm/sys Schema 与核心表建表脚本。  
4. 输出《01_系统架构设计》《02_数据库Schema设计》初稿。  
5. 增加健康检查接口 `/api/health`。  

**验收标准**：
- `uvicorn` 可启动；
- 数据库迁移成功；
- 健康检查可返回成功。

## 第 2 周：采集链路 + 数据质量
**目标**：打通 Tushare 到 dwd 的主链路。  
**任务**：
1. 完成股票池构建逻辑（创业板/科创板/科技行业）。  
2. 开发行情、复权、daily_basic、moneyflow、财务增量采集。  
3. 实现重试、限流、断点续跑机制。  
4. 建立数据质量检查（完整性、时效性、一致性）。  
5. 输出《03_Tushare数据采集接口契约》。

**验收标准**：
- 指定交易日采集成功率 ≥ 95%；
- 失败任务可自动重试并记录原因；
- 质量报告可查询。

## 第 3 周：因子评分 + 推荐产出
**目标**：实现“可解释评分 + 推荐输出”。  
**任务**：
1. 完成六大因子计算公式与标准化流程。

## 第 4 周：API 与前端联调

**任务**：
1. 开发推荐、个股、任务、因子、模拟相关 API。

**验收标准**：

## 6. 质量保障计划

### 6.1 测试分层
1. 单元测试：公式、过滤、手续费、排序逻辑。  
2. 集成测试：采集→入库→计算→推荐→复盘全链路。  
3. 回归测试：每次权重调整后执行 60/120 日对照测试。  

### 6.2 质量门禁
- 覆盖率目标：核心计算模块 ≥ 80%。  
- 接口 SLA：核心查询接口 P95 < 500ms。  
- 数据新鲜度：交易日 T 日 21:30 前完成全量计算。  
- 数据缺失阈值：核心字段缺失率 < 1%。

### 6.3 发布门禁
- 严重缺陷（P1）= 0。  
- 高风险缺陷（P2）已闭环。  
- 关键场景用例通过率 ≥ 95%。

## 7. 风险清单与应对

| 风险 | 影响 | 预警信号 | 应对策略 |
|---|---|---|---|
| Tushare 限频或接口波动 | 采集中断 | 连续失败率上升 | 增加重试、降频、分批采集 |
| 财务数据披露滞后 | 因子失真 | 近期财务缺失变高 | 使用公告日期回溯与缺失降权 |
| 因子过拟合 | 实盘失效 | Rank IC 快速衰减 | 滚动窗口验证 + 权重约束 |
| 小市值流动性不足 | 模拟偏乐观 | 成交额异常偏低 | 加强成交额过滤和冲击成本 |
| 调度任务冲突 | 结果不一致 | 任务重叠执行 | 单实例锁 + 幂等设计 |
| 部署环境差异 | 线上故障 | 预发通过、线上失败 | 建立环境基线与自动化检查 |

## 8. 运维与监控计划

### 8.1 监控项
- 进程可用性（FastAPI、Scheduler）。
- 数据采集成功率与耗时。
- 每日推荐数量异常波动。
- 复盘指标异常（回撤激增、胜率突降）。

### 8.2 告警等级
- P1：服务不可用、任务全失败。  
- P2：核心数据缺失、推荐为空。  
- P3：非核心任务失败、性能抖动。  

### 8.3 数据安全
- PostgreSQL 每日全量备份 + 每小时 WAL 增量。  
- 保留策略：全量 14 天，增量 7 天。  
- 每周执行一次恢复演练。

4. 具备后续版本平滑升级能力。
